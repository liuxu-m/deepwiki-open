# 项目列表备注与自动描述设计

## 背景

当前首页和 `/wiki/projects` 页面中的项目列表，仅展示以下信息：

- 项目名
- 平台（github / gitlab / bitbucket）
- 语言
- 处理时间

这能帮助用户找到已有 Wiki，但还不够表达“这个项目是做什么的”，也不支持用户对项目做个性化备注。因此希望在项目列表页面中，为每个项目增加：

- **自动描述**：告诉用户这个项目大概是做什么的
- **手动备注**：允许用户补充自己的说明

用户已经确认采用方案：**描述自动生成，备注手动填写**。

## 目标

- 在项目列表中为每个项目显示一段简短自动描述
- 支持用户为每个项目编辑并保存备注
- 自动描述随着 Wiki 内容变化而更新
- 手动备注独立存储，不受 Wiki 刷新覆盖
- 保持首页和 `/wiki/projects` 页面展示一致

## 非目标

- 不实现标签系统
- 不实现备注富文本
- 不实现备注搜索、排序、过滤
- 不实现多用户协作备注
- 不引入数据库表结构重构

## 现状总结

### 后端

- `/api/processed_projects` 会扫描 `wikicache` 目录下的缓存 JSON 文件
- 当前返回字段仅包含：
  - `id`
  - `owner`
  - `repo`
  - `name`
  - `repo_type`
  - `submittedAt`
  - `language`

### 前端

- 首页通过 `useProcessedProjects()` 获取项目列表
- `ProcessedProjects` 组件负责卡片/列表展示
- 当前 UI 中已有删除项目能力，但没有编辑元数据能力

## 方案选择

### 方案 A：描述和备注都手动填写

优点：实现简单。

缺点：
- 所有项目都需要人工维护
- 用户首次进入时信息为空
- 无法复用已有 Wiki 内容

### 方案 B：描述自动生成，备注手动填写（采用）

优点：
- 项目列表天然可读
- 备注与自动内容职责清晰
- 用户维护成本低
- Wiki 刷新后描述可自然更新

缺点：
- 后端需要增加摘要提取逻辑
- 需要新增元数据存储

### 方案 C：描述和备注都自动生成

优点：几乎零手工输入。

缺点：
- 备注失去“用户自己的语义”
- 无法表达“重点项目”“先别删”“客户项目”等私人标记

## 设计概览

### 自动描述

自动描述从现有 Wiki 缓存中提取，目标是给项目列表提供一句简短、稳定、可读的简介。

提取优先级建议如下：

1. 优先取 `wiki_structure.description`
2. 若为空，则尝试首页/概览页内容的首段
3. 若仍为空，则尝试首个已生成页面的开头内容
4. 如果都拿不到，则返回空值

为避免项目列表过长，描述建议限制在 **80~120 字** 左右，并去除 Markdown 标记。

### 手动备注

备注由用户维护，独立于 Wiki 缓存文件。

建议使用一个轻量的本地元数据文件保存，例如：

`~/.adalflow/project_metadata.json`

按项目唯一键保存：

`{repo_type}:{owner}:{repo}:{language}`

每条记录示例：

```json
{
  "github:AsyncFuncAI:deepwiki-open:zh": {
    "note": "重点关注后台任务队列实现",
    "updated_at": 1776600000000
  }
}
```

这样设计的好处是：
- 不污染 Wiki 缓存 JSON
- Wiki 刷新时不会覆盖备注
- 后续容易扩展收藏、标签、优先级等字段

## 后端设计

### 新增项目元数据读写能力

新增一个轻量模块，例如：

- `api/project_metadata.py`

职责：
- 生成项目唯一 key
- 读取 `project_metadata.json`
- 更新单个项目备注
- 删除项目时同步清理备注

### 扩展 `/api/processed_projects`

在现有返回结果中增加两个字段：

- `summary: string | null`
- `note: string | null`

处理流程：
1. 扫描每个 Wiki 缓存文件
2. 从缓存 JSON 中提取自动描述 `summary`
3. 用项目 key 查 `project_metadata.json`
4. 合并备注 `note`
5. 返回给前端

### 新增备注接口

建议新增：

- `PUT /api/processed_projects/{project_id}/note`

请求体示例：

```json
{
  "note": "这是客户项目，先别删"
}
```

行为：
- 根据 `project_id` 解析项目 key
- 更新或清空备注
- 返回更新后的 `note`

### 删除项目联动清理备注

现有删除项目接口在删除缓存文件时，应顺带删除该项目对应的备注元数据。

这样可以避免“项目删了，备注还残留”。

## 前端设计

### 列表展示

在 `ProcessedProjects` 卡片/列表项中新增两块信息：

1. **自动描述**
   - 放在项目标题下方
   - 样式比项目名更弱一些
   - 最多显示 2 行，超出省略

2. **备注**
   - 放在描述下方
   - 如果为空，显示“添加备注”按钮或占位提示
   - 如果有值，显示备注内容和“编辑”按钮

### 备注编辑交互

建议最小交互：

- 点击“添加备注”或“编辑备注”
- 当前卡片内展开一个小输入框
- 支持：
  - 保存
  - 取消
- 保存后即时更新当前列表项，无需整页刷新

备注字段建议限制长度，例如 **200 字以内**。

### 复用范围

`ProcessedProjects` 组件同时被首页和 `/wiki/projects` 页面复用，因此只需在该组件内统一实现，两个页面会自动获得新能力。

## 错误处理

### 自动描述提取失败

- 不影响项目列表主流程
- `summary` 返回空
- 前端不显示描述区域或显示“暂无描述”

### 备注保存失败

- 前端保留编辑框内容
- 提示“保存备注失败”
- 不影响其他项目展示

### 元数据文件损坏

- 后端应容错读取
- 读取失败时回退为空对象
- 写入时可重建文件

## 需要修改的文件

### 后端

- `api/api.py`
  - 扩展 `/api/processed_projects` 返回字段
  - 新增备注保存接口
  - 删除项目时联动清理备注

- `api/project_metadata.py`（新建）
  - 元数据文件读写与 key 管理

### 前端

- `src/components/ProcessedProjects.tsx`
  - 展示 `summary` 和 `note`
  - 增加备注编辑交互

- `src/hooks/useProcessedProjects.ts`（如该 hook 内有类型定义）
  - 扩展项目数据类型

- `src/types/...`（如项目类型单独定义）
  - 增加 `summary` / `note` 字段

## 测试策略

### 功能验证

1. 打开首页或 `/wiki/projects`
2. 确认每个项目显示自动描述（若缓存中可提取）
3. 点击“添加备注”
4. 输入备注并保存
5. 刷新页面
6. 确认备注仍存在

### 刷新 Wiki 验证

1. 给某项目写备注
2. 触发该项目的 Wiki 刷新
3. 刷新完成后重新打开项目列表
4. 确认自动描述可以更新
5. 确认备注不丢失

### 删除验证

1. 给某项目写备注
2. 删除该项目
3. 确认项目从列表中消失
4. 确认其备注元数据也被删除

## 成功标准

满足以下条件则认为功能完成：

- 项目列表可显示自动描述
- 用户可为项目保存备注
- 备注刷新页面后仍保留
- 刷新 Wiki 不会覆盖备注
- 删除项目时会清理对应备注
- 首页与 `/wiki/projects` 展示一致
